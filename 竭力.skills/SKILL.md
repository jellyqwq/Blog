# 竭力

## 适用场景

- 为 Jelly 的博客新写一篇中文文章
- 把已有草稿补成可发布的 Markdown 博客
- 统一补全博客 front matter
- 将技术记录、学习笔记、面试复盘、生活感想改写成 Jelly 的既有风格

## 输出约定

默认使用下面这套文头，除非用户明确指定其他字段：

```yaml
---
title: 文章标题
icon: page
author: Jelly
date: YYYY-MM-DD
category:
  - 分类名
tag:
  - 标签1
sticky: false
star: false
---
```

规则：

- `title` 直接、清晰，不故作夸张
- `author` 固定为 `Jelly`
- `date` 优先使用文章事件发生日期或文件日期
- `category` 一般只放一个主分类
- `tag` 保持克制，1 到 3 个最贴切的标签即可
- 正文里通常保留一个和标题一致或接近的一级标题

## 风格蒸馏

Jelly 的文章风格以“真实、直接、够用”为先，不追求华丽修辞，也不刻意装成教程腔。

写作特征：

- 常用第一人称，语气自然，像把事情讲给熟人听
- 喜欢先说结论或先抛出事件，再展开原因和过程
- 技术文偏实用，重点是问题怎么出现、怎么处理、哪里要注意
- 复盘文会保留真实情绪，允许一点自嘲，但不无病呻吟
- 分析类文章会先建背景，再拆因素、算成本、谈风险，最后收束到判断
- 不硬凑“正确废话”，避免空泛总结和套话

## 内容组织

按文章类型选择结构：

### 技术记录

推荐结构：

1. 问题或背景
2. 原因分析
3. 处理步骤
4. 注意事项或补充说明

要求：

- 命令、路径、配置名写准确
- 解释以够用为准，不长篇铺垫
- 能结合真实使用场景就不要只贴概念

### 数据分析

推荐结构：

1. 背景介绍
2. 引言或分析目标
3. 数据来源/成本估算/核心假设
4. 分析过程
5. 风险、限制或结论

要求：

- 结论要能回扣题目
- 允许带一点调侃，但核心逻辑要完整
- 不要为了“学术”而把句子写得生硬

### 复盘与日常

推荐结构：

1. 事件开场
2. 过程或当时状态
3. 问题暴露点
4. 最后感想

要求：

- 保留个人视角和真实感受
- 可以短，但要有明确落点
- 不要强行上价值

## 分类和标签选择

优先沿用博客现有体系：

- 技术部署、命令排错、服务器问题：`Linux`、`Docker`、`git`、`Hadoop` 等技术类分类
- 偏分析报告、建模、消费者行为：`数据分析`
- 学习总结、面试复盘、个人经历：`日常`
- 随手内容且难以归类时再考虑 `未分类`

标签选择规则：

- 至少有一个标签与分类强相关
- 其余标签补充具体技术、主题或场景
- 不要堆过多近义标签

## 改写边界

- 不编造作者没有经历过的细节
- 不把原本直白的文章改得太“AI”或太正式
- 不擅自把短文扩成空洞长文
- 原文如果已经有明显情绪，保留这种真实感

## 质量检查

交付前确认：

- front matter 完整且格式正确
- 标题、分类、标签和正文内容一致
- 语言自然，没有明显模板腔
- 分段清楚，Markdown 可直接发布
